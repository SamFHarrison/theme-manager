# `useTheme()`: Explanation

The `useTheme()` hook provides a small theme-management layer for React applications. It connects three sources of theme information:

- the user's saved preference in `localStorage`,
- the theme marker applied to the root `<html>` element,
- the browser or operating system `prefers-color-scheme` setting.

The hook returns both the user's preference and the theme that is currently resolved from that preference.

```ts
const { preferredTheme, resolvedTheme, setTheme } = useTheme();
```

## Preferred theme vs resolved theme

The hook distinguishes between two related ideas: `preferredTheme` and `resolvedTheme`.

`preferredTheme` is the value the user has chosen. It's type is...

```ts
type ThemePreference = 'auto' | 'dark' | 'light';
```

`resolvedTheme` is the actual theme currently in effect. It's type is...

```ts
type ResolvedTheme = 'dark' | 'light';
```

The distinction matters because `auto` is not a visual theme. It means "follow the user's system setting". So `auto` must resolve to either `dark` or `light`.

For example, if the preferred theme is `auto` and the user's system preference is dark, the resolved theme is `dark`. If the system preference is light, the resolved theme is `light`.

## Why the root element is the source of truth

The hook resolves the active theme by looking at the marker on `document.documentElement`, which is the root `<html>` element.

In the default class mode, the rules are:

| Root class state               | Meaning                    |
| ------------------------------ | -------------------------- |
| `.obds-color-pref-dark` exists | Force dark mode            |
| `.allow-prefers-dark` exists   | Use `prefers-color-scheme` |
| neither class exists           | Use light mode             |

In attribute mode, the root element is instead updated with an attribute such as `data-theme="dark"` or `data-theme="light"`.

The hook intentionally treats the root marker as the source of truth for the current rendered theme. This is useful because CSS and design-system styles usually respond to that root state directly.

It also means the hook remains accurate if something outside React changes the root marker. For example, an inline bootstrap script, legacy code, a test helper, or another part of the app can update the root state and the hook will still reflect the actual theme being applied.

## How the default works

If no value exists in `localStorage` under the `hbo-theme-preference` key, the hook defaults the preference to `auto`.

This means the app respects the user's browser or operating system colour-scheme preference by default.

The default behaviour is:

1. Try to read `hbo-theme-preference` from `localStorage`.
2. If the value is `auto`, `dark`, or `light`, use it.
3. If the value is missing or invalid, save `auto`.
4. Apply the default root theme marker.
5. Resolve the actual theme from `prefers-color-scheme`.

## Why `useSyncExternalStore` is used

The hook uses React's `useSyncExternalStore` because the theme state does not live purely inside React.

The state is spread across external browser features:

- `localStorage`,
- `document.documentElement.classList`,
- `window.matchMedia('(prefers-color-scheme: dark)')`.

React does not automatically know when those things change. `useSyncExternalStore` lets the hook subscribe to external changes and tell React when to re-read the current snapshot.

The hook subscribes to:

- the native `storage` event, for changes from other tabs or windows,
- a custom `hbo-theme-preference-change` event, for changes made in the current tab,
- the `prefers-color-scheme` media query, for system theme changes,
- a `MutationObserver`, for root marker changes.

## Why the snapshot is a string

The hook uses this type:

```ts
type ThemeSnapshot = `${ThemePreference}:${ResolvedTheme}`;
```

The snapshot is a string such as:

```ts
'auto:dark';
'dark:dark';
'light:light';
'auto:light';
```

This is deliberate. `useSyncExternalStore` expects the snapshot to be referentially stable when nothing has changed.

If the hook returned an object like this from `getSnapshot()`:

```ts
{
  preferredTheme: 'auto',
  resolvedTheme: 'dark'
}
```

then a new object would be created every time, even if the values were the same. That can cause unnecessary renders and React warnings.

A string is a primitive value, so React can compare it safely.

## How updates stay in sync

The hook keeps theme values up-to-date across several scenarios.

When `setTheme()` is called in the current tab, it writes to `localStorage`, applies the root classes, and dispatches the custom change event. That custom event tells React subscribers in the same tab to update.

When the theme is changed in another tab, the browser's native `storage` event fires. The hook listens for that event and re-reads the snapshot.

When the system colour scheme changes, the `prefers-color-scheme` media query fires a `change` event. This matters when the preference is `auto`.

When something outside the hook changes the root marker, the `MutationObserver` sees the mutation and asks React to re-read the snapshot.

`localStorage` was a conscious decision to enable this multi-session and multi-tab consistency that `sessionStorage` wouldn't allow us to do.

## Why storage helpers fail quietly

The hook wraps `localStorage` reads and writes in safe helper functions.

This is because `window` can exist while `localStorage` is still inaccessible. That can happen in some privacy modes, restricted browser contexts, embedded frames, enterprise-managed browsers, or test environments.

The safe getter returns `null` if storage cannot be read. The safe setter returns `false` if storage cannot be written.

Storage read failures are treated as a fallback case. The hook still applies the default `auto` behaviour for the current page session.

Storage write failures during `setTheme()` do not stop the visual theme from updating. The theme class is still applied, but the preference may not persist after reload or in other sessions.

## Current trade-offs

This implementation is intentionally pragmatic.

`getSnapshot()` can indirectly initialise the default preference because it calls `getPreferredTheme()`. That means it can write `auto` to storage and apply the root class during a read. In strict terms, this is a side effect inside snapshot calculation.

The upside is that missing or invalid state is corrected early, and the app gets to the right theme state quickly.

A stricter alternative would move initialisation to an effect or a bootstrap script. That would make the snapshot read purer, but it could delay class application and increase the chance of a flash of the wrong theme.

For this hook, the pragmatic approach is reasonable because theme initialisation is most useful when it happens early.
